# Database Schema Documentation: Soldier-Host Matching System

This document outlines the database schema designed to support the matching platform connecting soldiers with host families for meals and weekend stays.

---

## 1. `soldiers` (Soldier Users)
Stores the profile, preferences, and permanent restrictions of soldiers.

| Field Name | Data Type | Description / Constraints |
| :--- | :--- | :--- |
| `id` | `uuid` | Primary Key |
| `created_at` | `date` | Account creation timestamp |
| `full_name` | `text` | |
| `phone` | `text` | |
| `age` | `int` | |
| `profile_img_url` | `text` | URL to profile picture |
| `bio` | `text` | Short biography |
| `kosher_level` | `text` | Options: `kosher_strictly`, `kosher`, `separation`, `none` |
| `shabbat_keeping` | `bool` | `true` if they strictly observe Shabbat |
| `pet_compatibility` | `text` | Options: `comfortable`, `uncomfortable`, `allergic` |
| `allergies_and_diet` | `list` | Elements: `gluten_free`, `vegan`, `vegetarian`, etc. |
| `languages` | `list` | Elements: `hebrew`, `english`, `russian`, etc. |
| `banned_families` | `list` / `uuid[]` | Global blocklist: Family IDs this soldier permanently refuses to stay with |

## 3. `soldier_hosting_searches` (Soldier Hosting Requests)
Represents a specific search instance initiated by a soldier for an upcoming meal or weekend.

| Field Name | Data Type | Description / Constraints |
| :--- | :--- | :--- |
| `id` | `uuid` | Primary Key |
| `soldier_id` | `uuid` | Foreign Key (references `soldiers.id`) |
| `dinner_date` | `date` | Target date of the event |
| `arrival_time_range` | `text` | e.g., "17:00-19:00" |
| `num_of_guests` | `int` | |
| `additional_guests_preferences` | `json` | Holds dietary/pet restrictions if traveling with friends |
| `stay_duration` | `text` | e.g., "Dinner only", "All weekend" |
| `needs_transportation` | `bool` | |
| `location_lat` | `float` | Latitude of search origin/center |
| `location_lng` | `float` | Longitude of search origin/center |
| `search_radius_km` | `float` | Travel distance tolerance |
| `notes` | `text` | Special messages or custom requests |
| `is_match` | `bool` | Flag indicating if request has been fulfilled |
| `temporarily_banned_families` | `list` / `uuid[]` | Excluded Family IDs **only for this specific search request** |

## 5. `active_matches`
Tracks proposals and finalized connections generated by the matching pipeline.

| Field Name | Data Type | Description / Lifecycle |
| :--- | :--- | :--- |
| `id` | `uuid` | Primary Key |
| `created_at` | `date` | Timestamp of proposal generation |
| `soldier_request_id` | `uuid` | Foreign Key (references `soldier_hosting_searches.id`) |
| `host_offer_id` | `uuid` | Foreign Key (references `family_hostings.id`) |
| `status` | `text` | Options:<br>• `approved` (Initial default state: system matched)<br>• `rejected` (Declined; triggers engine to recalculate) |
| `distance_km` | `number` | Travel distance (km) between the soldier's chosen location and the family. Computed by Google Distance Matrix, falling back to straight-line. `null` when coordinates are unavailable. |

## 💡 Matching Algorithm Rule Note
When querying potential matches for a given `soldier_hosting_searches` record, the engine must drop hosts whose `family_id` is found within:
1. `soldiers.banned_families` (Permanent account-level exclusion)
2. `soldier_hosting_searches.temporarily_banned_families` (Per-request exclusion)

**Location matching:** after the hard filters run, the engine measures the
distance from the request's `lat`/`lng` to each candidate family's
`hostLat`/`hostLng`. Real road/walking distance comes from Google Distance
Matrix (one batched call per request, walking mode when the soldier needs
walking distance), falling back to straight-line haversine if the API or
coordinates are unavailable. Families beyond the request's `travelDistance`
(widened 20% under the RADIUS compromise) are dropped, and the remaining
families receive a proximity bonus so the **closest** acceptable family wins.
